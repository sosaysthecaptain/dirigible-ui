How to Use

  1. Start with the token preamble from the Tokens page — copy the :root block into your prompt once per session
  2. Find the component you need in the sidebar
  3. Copy the recipe text for the variant you want
  4. Paste into Claude Code alongside your layout requirements
  5. Combine recipes freely: "Use modal-confirm with button-danger and input-text"
  6. Reference token names (var(--fg)), not hex values, for automatic theme support

Recipes

Recipes are terse CSS property specifications. Each one describes a single component variant with exact values for font, padding, border, color, and interactive states. No ambiguity — every property is explicit. Copy a recipe, paste it into your prompt, and get the exact component. For example:

button-primary
  font: 600 13px/20px inherit
  padding: 2px 12px
  background: var(--fg); color: var(--bg)
  border: 3px solid var(--fg); border-radius: 4px
  :hover -> opacity: 0.85
  :active -> background: var(--bg); color: var(--fg)

Layer Model

0 Tokens CSS custom properties — --bg, --fg, --border, --accent, --radius-*
1 Atoms Single-element components — button, input, toggle, badge, slider, toast
2 Composites Multi-element assemblies — modal, menu, table, popover, select
3 Catalog Reference lookups — icons, theme colors

Always include the token preamble (Layer 0). Atoms are self-contained. Composites compose atoms — a modal uses button recipes, a popover uses input recipes.

Philosophy

The UI is quiet, monospaced, and warm. It looks like a tool a thoughtful person built for themselves. Every element earns its place — no decorative flourishes, no gratuitous color, no rounded-everything friendliness. The aesthetic sits somewhere between a terminal and a well-typeset book.

Core Principles

  1. Monospace-first. SF Mono is the default for everything — sidebar, buttons, table cells, form labels. Serif and sans options exist for content, but the app itself speaks in monospace.
  2. Warm neutrals. The light theme is cream, not white. The dark theme is brown, not gray. Borders are soft (#e8e6e3), not clinical (#e0e0e0). This warmth is the single most distinctive visual quality.
  3. Sparse color. Accent color (#3d8ad3, a muted blue) is used sparingly — active tabs, primary actions, status dots. Most of the UI is foreground-on-background with muted secondary text. Color means something happened.
  4. 2px borders on interactive elements. Buttons, inputs, toggles — anything you can click gets a 2px solid border. Dividers and table rules get 1px. This creates a clear visual distinction between "things" and "structure."
  5. Information density. Tables show everything. Density is a feature, not a bug.
  6. Semantic headings only. Headings use weight (semibold) and size. No text-transform, no letter-spacing tricks, no uppercase labels.

Typography

Dirigible uses four font stacks, each with a specific role:

  • Monospace — 'SF Mono', 'Cascadia Code', 'Fira Code', 'JetBrains Mono', monospace — The default for everything. UI chrome, buttons, labels, form inputs, navigation, code. All headings use semibold monospace. This is the voice of the application.
  • Sans-serif — 'IBM Plex Sans', -apple-system, sans-serif — Body text in documents and notes when the user selects sans mode. Never used for UI chrome.
  • Serif — 'Crimson Pro', Georgia, serif — Body text in documents and notes when the user selects serif mode. Gives a literary quality to long-form writing.
  • System — 'Lucida Grande', 'Lucida Sans Unicode', 'Helvetica Neue', sans-serif — Used exclusively for data tables. Maximum density and legibility at 10–11px.
