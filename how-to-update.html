How to Update

dirigible-ui is designed to be maintained by AI coding assistants. You describe what you want changed in natural language, Claude (or similar) edits the HTML files directly, and you review the result in a browser. There is no build step.

Philosophy

Every page in dirigible-ui is a standalone HTML file with inline styles and live interactive demos. This is deliberate:

  • No abstraction layer. There's no Sass, no CSS-in-JS, no component library to compile. The HTML files are both the source and the output. What you see in the browser is what Claude reads when building your app.
  • AI-native authoring. These files are written to be read and modified by language models. The inline styles, the detailed spec tables, the code examples — they're all structured so an AI can parse them reliably and produce consistent output.
  • Human-reviewable. Open a file in a browser, look at it, decide if it's right. No build logs to parse, no dev server to start. The feedback loop is: describe a change → AI edits → refresh browser → approve or iterate.
  • Single source of truth. The live demo and the spec text live in the same file. They can't drift apart because there's nothing to keep in sync — it's all one document.

Tweaking a Component

To change how a component looks or behaves, tell your AI assistant what you want. Be specific about the visual change:

# Example prompts:
"Make primary buttons use a 2px border instead of 3px"
"Change the accent color to #E8915A across all files"
"Add a hover state to the badge component — subtle background fill"
"The modal tabs should use the accent color for the active underline, not foreground"

The AI will:

  1. Read the relevant component file (e.g., components/button.html)
  2. Edit the inline styles in the live render demo
  3. Update the spec tables, CSS examples, and rules to match
  4. Optionally propagate the change to other files if it touches shared values (like accent color)

After the edit, open the file in a browser and verify visually. If it's not right, describe what's wrong and iterate.

Changing Design Tokens

Design tokens (colors, spacing, typography) are defined as CSS custom properties in the :root block at the top of every HTML file. To change a token globally:

# Tell your AI:
"Change --accent from #3d8ad3 to #2563eb across all HTML files in dirigible-ui"

The AI will find-and-replace across all 30+ files. It should also update any documentation text that references the old value (like the overview page mentioning the accent color).

Token changes that affect the token documentation pages (tokens/colors.html, etc.) should update both the CSS variables and the spec content on those pages.

Adding a New Component

# Tell your AI:
"Add a new component page for a dropdown/select input.
Follow the same HTML template as components/button.html —
same head, sidebar nav, style block, theme toggle.
Include a live render demo showing the dropdown in various states,
then spec tables, code examples, and do's/don'ts."

The AI should also add the new page to the sidebar nav in every existing HTML file.

Updating a Project Built on This System

When the design system changes, projects consuming it need to be updated. The process:

  1. Pull the latest design system — git submodule update --remote dirigible-ui (or git pull if it's a sibling clone)
  2. Tell your AI to audit — give it a prompt like:
"The design system in ./dirigible-ui/ has been updated.
Read the changed component files and audit our codebase
for any UI that no longer matches. Fix what's drifted."

The AI will diff the new specs against your implementation and make targeted fixes. This is much faster than manually auditing because the AI can read every component spec and cross-reference it against your actual code.

Sidebar Nav Maintenance

Every HTML file contains its own copy of the sidebar navigation. When you add or rename a page, the nav needs updating in all files. Tell the AI:

"Add a nav link for 'Dropdown' in the Components section of the sidebar,
in every HTML file in dirigible-ui. It should go after 'Context Menu'
alphabetically."

What Not to Do

  • Don't add a build step. The power of this system is its simplicity. The moment you introduce a preprocessor, a template engine, or a component framework, you lose the directness that makes AI-assisted editing work so well.
  • Don't extract shared styles. Yes, the CSS is duplicated across files. That's the point. Each file is self-contained and independently readable. Shared stylesheets create coupling and make AI edits harder to reason about.
  • Don't edit only the demo or only the spec. When changing a component, both the live render and the spec text must be updated together. They're in the same file for exactly this reason.
