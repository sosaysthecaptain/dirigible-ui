How to Use

dirigible-ui is an agent-first UI framework that serves two audiences simultaneously: humans who open it in a browser, and AI coding assistants (Claude, Cursor, etc.) that read the raw HTML files as specifications.

Including in a Project

Add dirigible-ui as a git submodule in any project that needs to follow the design system:

git submodule add https://github.com/sosaysthecaptain/dirigible-ui.git dirigible-ui

This keeps your copy linked to the upstream repo. When the design system is updated, pull the latest:

git submodule update --remote dirigible-ui

Alternatively, clone it as a sibling directory if you prefer not to nest it inside your project.

Pointing Claude at It

Add this to your project's CLAUDE.md:

## Design System
Follow the design system in ./dirigible-ui/
Before creating or modifying any UI component, read the relevant
component and token files in ./dirigible-ui/tokens/ and ./dirigible-ui/components/.
Use CSS custom properties — never raw hex values.
Match the patterns, spacing, and interactive behaviors documented
in the component files exactly.

When Claude Code sees this instruction, it will read the relevant .html files before generating UI. Each file contains the full spec — tokens, variants, states, hover behaviors — plus live inline demos, all parseable from raw HTML.

What Claude Reads

Each component page is a self-contained HTML file with:

  • CSS custom properties in the :root block — Claude extracts the exact color values, spacing tokens, and semantic variable names
  • Live render demos with inline styles — Claude sees the precise styling applied to each element: border widths, padding values, font weights, hover behaviors
  • Spec tables documenting variants, states, and when to use each
  • Code examples in CSS, HTML, and JSX showing the expected implementation patterns
  • Do's and don'ts that constrain Claude's output to match the design system

For example, if you ask Claude to "add a delete confirmation modal," it reads components/modal.html and components/button.html, sees that modals have no separate footer, that primary buttons use 3px borders with bold text, that danger buttons use red text with a subtle hover tint — and generates code matching all of that.

Browsing as a Human

Open any .html file directly in a browser. No build step, no server, no dependencies. The sidebar nav links everything together. Use the moon/sun toggle in the sidebar header to switch between light and dark themes.

What to Read First

  1. Colors — understand the palette and semantic token names
  2. Typography — font stacks, scale, and rules
  3. Button — the most opinionated component, sets the tone for everything
  4. Modal — shows how multiple components compose together
  5. Music Table — a full-page example showing information density in practice
